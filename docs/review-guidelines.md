# 리뷰 규칙

> Codex GitHub review, Claude Code, Codex CLI/App이 PR 리뷰를 할 때 공통으로 따르는 기준.

## 기본 원칙

- 리뷰 코멘트는 항상 한국어로 작성한다.
- 리뷰는 취향이 아니라 실제 위험을 기준으로 작성한다.
- 가장 중요한 문제부터 남긴다. 사소한 스타일 선호는 리뷰 코멘트로 남기지 않는다.
- 변경 diff만 보지 말고, 관련 파일의 기존 구조와 프로젝트 문서를 함께 고려한다.
- 확실하지 않은 내용은 단정하지 말고 확인이 필요한 질문으로 남긴다.

## 우선순위

1. 런타임 버그, 빌드 실패, 타입 오류
2. 사용자 흐름이 깨지는 회귀
3. 보안, 인증, API 키, 개인정보 노출 위험
4. 데이터 손실 또는 Supabase/TMDB/OpenAI 호출 구조 문제
5. 누락된 테스트 또는 검증 부족
6. 유지보수에 직접 영향을 주는 구조 문제

## 코멘트 작성 방식

- 문제 위치를 파일과 라인 기준으로 명확히 짚는다.
- 왜 문제인지, 어떤 영향이 있는지, 어떻게 고치면 되는지 함께 설명한다.
- 한 코멘트에는 한 가지 문제만 다룬다.
- 코드 작성자의 의도를 추측해 비난하지 않는다.
- 단순 문구, 공백, 취향 차이는 기능이나 유지보수 위험이 있을 때만 언급한다.

## 이 저장소에서 특히 볼 것

- `App`은 라우터 렌더링만 담당하고, 공통 UI는 layout 컴포넌트에서 관리하는지 확인한다.
- `Outlet`은 layout 계층에서만 사용한다.
- `src/components/ui`는 shadcn/ui 자동 생성 영역이므로 직접 수정하지 않는다.
- API 키나 OpenAI 키가 클라이언트 코드와 문서에 노출되지 않았는지 확인한다.
- 기능 추가와 리팩터링이 한 PR에 섞이지 않았는지 확인한다.
- 문서와 실제 코드가 불일치하면, 커밋을 막는 대신 PR 리뷰나 후속 작업으로 명확히 남긴다.

## 리뷰하지 않아도 되는 것

- 단순 취향의 클래스 순서
- 아직 확정되지 않은 디자인 토큰 네이밍
- 기능 위험이 없는 문장 표현 차이
- 후속 작업으로 명시된 미구현 범위
