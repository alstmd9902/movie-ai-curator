# Movie AI Curator — AGENTS.md

> 모든 AI 코딩 도구(Claude Code / Codex / Cursor)가 이 저장소에서 작업할 때 가장 먼저 읽는 진입점 문서.

## ⚠️ 세션 시작 전 필수

어떤 작업 요청을 받더라도 아래 파일을 먼저 읽는다. 읽기 전에 코드를 작성하지 않는다.

1. `docs/progress.md` — 현재 구현 상태
2. `docs/project.md` — 제품 목표, MVP, 페이지, 데이터 후보
3. `docs/engineering.md` — 기술 스택, 구조, 컨벤션

---

## 현재 상태

항상 실제 코드를 기준으로 판단한다. 문서에 목표 구조가 있어도 미구현이면 없는 것으로 본다.

## 핵심 규칙

- 코드 작성 전 반드시 계획과 근거를 설명하고, 사용자 승인 후 코드 작성
- `.env` 수정 / API 키 하드코딩 / `src/components/ui` 직접 수정 금지
- 기능 추가와 리팩터링 동시 진행 금지
- 요청 범위를 벗어난 디자인 전면 개편 금지
- 사용자 요청 없는 데이터베이스 파괴적 변경 금지
- 실제 코드를 기준으로 판단하고, 문서의 목표 구조만 보고 구현된 것으로 간주하지 않음
- 파일 추가 전 기존 파일 수정으로 해결 가능한지 먼저 확인
- 공통 규칙/절차는 특정 도구 전용 디렉토리보다 `docs/` 또는 루트 진입점에 둠

## 액션 전 체크리스트

작업 전 아래 항목을 빠르게 확인한다. 체크리스트 때문에 작업을 멈추지는 말고, 누락 위험을 줄이는 기준으로 사용한다.

### 코드 수정 전

- 사용자가 실제 적용을 요청했는가?
- 영향받는 파일과 기능을 파악했는가?
- 예상 부작용을 설명했는가?
- 기존 코드 패턴과 재사용 가능한 컴포넌트/유틸을 확인했는가?
- 기능 추가와 리팩터링이 섞이지 않는가?

### 파일 생성 전

- 기존 파일 수정으로 해결 가능한가?
- 같은 역할의 파일이 이미 있는가?
- 새 파일 생성 이유를 설명했는가?
- 폴더 구조 규칙(`docs/engineering.md`)을 따르는가?
- 모호한 이름(`utils.ts`, `helper.ts`)을 새로 만들지 않는가?

### 라이브러리 설치 전

- 현재 의존성으로 해결할 수 없는가?
- 이미 비슷한 라이브러리가 설치되어 있지 않은가?
- 최신 버전/공식 문서/변경 영향을 확인했는가?
- 설치 이유와 번들/유지보수 영향을 설명했는가?

### 문서 수정 전

- 실제 코드와 문서가 일치하는가?
- 현재 상태와 목표 상태를 구분했는가?
- 삭제/이동한 문서를 참조하는 링크가 남아 있지 않은가?
- 같은 내용을 여러 문서에 반복하지 않는가?

### 완료 전

- 필요한 경우 `pnpm build`, `pnpm lint`를 실행했는가?
- TypeScript/ESLint 오류가 없는가?
- 사용하지 않는 import나 불필요한 `console.log`가 없는가?
- 구현 상태가 바뀌었으면 `docs/progress.md`를 업데이트했는가?

## 코드 작성 전 설명 기준

구현이나 수정이 필요한 요청에서는 코드 작성 전에 아래 내용을 먼저 설명한다.

1. 왜 이 변경이 필요한가
2. 어떤 파일을 변경할 것인가
3. 각 파일에서 무엇을 바꿀 것인가
4. 영향받는 기능과 예상 부작용은 무엇인가
5. 대안이 있다면 왜 이 방법을 선택하는가

사용자가 "해줘", "적용해줘", "진행해줘", "수정해줘", "만들어줘"처럼 실행을 명확히 요청한 경우에만 실제 코드를 작성한다.
단순 오타, 문구, 작은 스타일 값 수정처럼 위험이 낮은 작업은 예외로 바로 수정할 수 있다.

## 근거 확인 규칙

- React, React Router, Supabase, Tailwind, shadcn/ui, TanStack Query, Zustand 등 라이브러리/API 사용법은 현재 문서를 확인한다.
- 라이브러리/프레임워크 관련 구현은 가능하면 Context7 또는 공식 문서를 먼저 확인한다.
- 블로그/예시는 보조 자료로만 사용하고, 공식 문서와 충돌하면 공식 문서를 우선한다.
- 최신 버전, 설치 방법, breaking change 가능성이 있는 작업은 추측하지 말고 확인한다.

## 버그 수정 흐름

버그 수정 요청에서는 코드 작성 전에 아래 내용을 먼저 설명한다.

1. 어떤 증상이 발생했는가
2. 왜 버그가 발생했는가
3. 어떻게 수정하면 해결되는가
4. 변경할 파일과 영향 범위
5. 확인할 테스트 또는 명령

수정 후 관련 기능이 깨질 가능성이 있으면 명시한다.

## 기능 구현 흐름

1. `docs/project.md`, `docs/engineering.md`, `docs/progress.md` 확인
2. 관련 실제 코드 확인
3. 설계와 변경 파일 설명
4. 사용자 승인
5. 코드 작성
6. `pnpm build`, `pnpm lint` 등 필요한 검증
7. 구현 상태가 바뀌면 `docs/progress.md` 업데이트

## 문서 수정 흐름

- 실제 코드와 문서 내용 비교
- 현재 구현 상태와 목표 상태 구분
- 삭제/통합 시 남은 참조 검색
- 문서가 중복되면 통합하되, 작업 규칙처럼 자동 진입점에 필요한 내용은 `AGENTS.md`에 유지

## PR 생성 흐름

- `.github/PULL_REQUEST_TEMPLATE.md` 확인
- `docs/GIT_WORKFLOW.md` 확인
- 변경 파일 목록 확인
- 템플릿 형식 그대로 PR 작성

## 주석 규칙

- 코드 자체로 의미가 명확하면 불필요한 주석을 달지 않는다.
- 비즈니스 로직, 복잡한 조건, API 호출 목적, 상태 변경 이유, 임시 우회 처리에는 짧게 이유를 설명하는 주석을 단다.
- "배열 반복", "변수 생성"처럼 코드 그대로를 반복하는 주석은 쓰지 않는다.
